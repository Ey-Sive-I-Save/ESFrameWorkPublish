以下为最终目标

我将会基于Playable开发动画系统，具有的特性的参数“代价”化，多态序列化，支持多层级，IK混合，Buff，可重用性，有以下特征:
1.预先定义几条流水线，比如基本线(控制跑跳下蹲的硬性过渡支撑动画)，主线(控制技能，表情，交互这种互相排斥的动画和执行)，Buff线(控制Buff，这种不一定输出动作，而是执行效果)。

三条线会稳定Mix输出到Playable

2.代价参数，人的四肢，意愿等都会编成代价值(浮点数)，一个动作进入时需要消耗代价，进入后摇阶段会逐步释放代价(退出时必须完全返还)。 

 当进入一个新主线动作时，如果代价条件符合就可以直接进入(并且占据同线)，如果不符合则开始测试，先过主线测试(一般不会通过)，通过的话再做基本和Buff测试，最后收集打断目标并且完成过渡，否则直接忽略掉跳转并且更新备忘状态。

3.同路，静止→走路→奔跑是同备忘状态路动画，特点是被弱打断时不会直接退出，而是往低级退化。

4.备忘状态  为了防止合并冲突测试每帧发生并且损耗过大，只有当整个状态机发生过状态退出，退化或者后摇，才会刷新进入备忘状态，否则会滞留“尝试进入”→“禁止”的列表。如果不参与代价计算，一般不会影响备忘状态。

5.上下文参数，自由加入的可变参数，一般用于制作硬性进入和退出条件判据，和原生控制器
一样。
↑浮点，整数，触发器，状态值↑实体，字符串标记，临时代价


6.曲线也是上下文的一种，一般被动画控制，完成绑定可以自动处理对应IK。

7.Clip生成单独的配置表，不内置在动画中，而是按键查，方便复用替换

8.每个组分都是单独的多态序列化对象，比如显示组分，过渡组分，执行组分…

9.总状态机单独拥有一个SO数据，但是和Clip替换拆分开，并且可以作为默认“环境”来提供Clip表和参数等数据。总状态机和基本线通常直接包含，但是主线和Buff线都是动态装载

10.显示器不一定直接绑定一个Clip，可能把Clip截断或者多个Clip阶段并组合
