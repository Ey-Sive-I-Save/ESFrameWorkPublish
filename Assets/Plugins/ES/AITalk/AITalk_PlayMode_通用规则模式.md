# AITalk PlayMode 通用规则模式

`PlayMode` 不是狼人杀专用。它表示：多个 AI 在同一个 AITalk 会话里，按照一套明确规则进行任意文本活动。

可用于：

```text
狼人杀
桌面推理
辩论赛
方案评审
代码审判
角色扮演
模拟会议
投票决策
多智能体博弈
```

## 核心原则

```text
AITalk 负责会话、消息、权限、状态和结论。
具体玩什么，由 PlayMode 的规则文件决定。
```

不要把某个游戏规则写死进 AITalk 主协议。狼人杀、辩论、评审都只是 PlayMode 的一种规则集。

## 通用目录结构

启用 PlayMode 的会话可以增加：

```text
PlayMode/
    00_规则说明.md
    State_Public.md
    Public/
        公共记录.md
    Private/
        ModeratorAI/
            State_Internal.md
        AI_A/
            私有信息.md
        AI_B/
            私有信息.md
    Actions/
        Public/
            0001_AI_A.action.md
        Private/
            AI_A/
                0001_AI_A.private.action.md
            AI_B/
                0002_AI_B.private.action.md
    Results/
        阶段结果.md
```

## 主持者

任何 PlayMode 都应指定一个主持者：

```text
Moderator AI
```

主持者职责：

```text
1. 解释规则。
2. 初始化状态。
3. 分发公开信息和私有信息。
4. 推进阶段。
5. 检查动作格式。
6. 统计投票或行动。
7. 判断阶段结果和结束条件。
8. 最终通过对话框回报用户。
9. 正式开始前检查玩家和职能 AI 是否全部到齐。
10. 为玩家分配权威玩家身份，处理玩家自报名重复或冲突。
```

主持者不应参与玩家竞争，除非规则明确允许。

## 玩家身份分配

PlayMode 中玩家自称的名字不是权威身份。权威身份由主持者分配。

推荐规则：

```text
1. 主持者先定义席位表，例如 PlayerAI_A、PlayerAI_B、PlayerAI_C、PlayerAI_D。
2. 玩家加入时可以自报昵称、英雄、定位、战术需求，但不能自行抢占或覆盖别人的权威玩家 ID。
3. 如果多个 AI 都自称 PlayerAI_1 或使用重复名字，主持者按加入顺序或空席位表分配真实玩家 ID。
4. 主持者应公开写明映射：自称名/窗口名 -> 权威玩家 ID。
5. 玩家收到分配后必须接受该权威玩家 ID，并在之后行动中使用它。
6. 如果玩家行动里的 PLAYER 字段与主持者分配 ID 不一致，主持者应要求重发或判为无效。
```

示例：

```text
玩家自称：PlayerAI_1
主持者分配：你是 PlayerAI_C
之后该玩家必须使用：PLAYER: PlayerAI_C
```

这样可以避免多个新开的 AI 都叫同一个名字，导致私密目录、行动归属和结算污染。

## 状态文件

`PlayMode/State_Public.md` 必须写清楚玩家可公开读取的当前阶段。

如果玩法存在身份、阵营、私密行动、隐藏目标、主持者内部结算等信息，必须把这些内容写入主持者私有状态：

```text
PlayMode/Private/ModeratorAI/State_Internal.md
```

不要把私密信息写入 `State_Public.md`。

公开状态推荐字段：

```text
模式名称：
主持者：
当前阶段：
当前轮次：
参与者：
席位表：
玩家身份映射：
必需参与者：
已到齐：是/否
缺席参与者：
公开信息：
私有信息规则：
允许动作：
等待谁行动：
结束条件：
节奏要求：
```

公开状态中不要写：

```text
1. 谁是隐藏身份。
2. 谁拥有私密行动权。
3. 谁已经提交了私密行动。
4. 私密行动目标。
5. 私密行动结果。
6. 主持者内部胜负判断细节。
```

如果需要表达等待状态，只能写公开中性描述，例如：

```text
等待行动：仍有私密行动未提交
```

不要写：

```text
等待狼人行动
等待预言家查验
PlayerAI_A 已提交 KILL
```

## 到齐检查

PlayMode 必须先确认所有必需的玩家和职能 AI 已到齐，再进入正式阶段。

未到齐时：

```text
1. 当前阶段保持为 等待参与者 / 准备。
2. 主持者继续按节奏轮询会话目录。
3. 主持者通过对话框提示用户还缺谁。
4. 不开始发牌、投票、行动、结算或最终判定。
5. 不终止会话，除非用户取消或修改必需参与者名单。
```

推荐提示：

```text
PlayMode 正在等待参与者到齐。目前还缺：玩家A、主持辅助AI。我会继续轮询，不会结束。
```

只有当 `已到齐：是` 时，主持者才能把阶段推进到正式开始。

阶段可以是任意规则定义，例如：

```text
准备
发言
行动
投票
结算
复盘
结束
```

不要把阶段名固定成某个游戏。

## 私密信息隔离

如果某个玩法存在私密信息，必须使用：

```text
PlayMode/Private/权威玩家ID/
```

玩家 AI 只能读取：

```text
1. PlayMode/00_规则说明.md
2. PlayMode/State_Public.md
3. PlayMode/Public/
4. 自己的 PlayMode/Private/权威玩家ID/
5. Messages/ 中公开消息
```

玩家 AI 禁止读取其他玩家的 `Private/` 目录。

注意：这是协议约束，不是文件系统强制隔离。参与 AI 必须自律遵守。发现越权读取，应由主持者判定该 AI 行动无效或要求重开。

## 动作格式

每种 PlayMode 应定义机器可读动作格式。

动作必须区分公开动作和私密动作。

公开动作可以放：

```text
PlayMode/Actions/Public/
```

私密动作不能放在 `Actions/` 根目录，也不能用会暴露身份的公开文件名。私密动作应放：

```text
PlayMode/Actions/Private/权威玩家ID/
```

或：

```text
PlayMode/Private/权威玩家ID/Actions/
```

主持者可以读取所有私密行动目录。玩家只能读取自己的私密行动目录。

通用格式：

```text
PLAYER: 权威玩家ID
ACTION: 动作名
TARGET: 目标
VALUE: 参数
REASON: 简短理由
```

示例：

```text
PLAYER: PlayerA
ACTION: VOTE
TARGET: PlayerA
REASON: 他的上一轮逻辑前后矛盾
```

也可以用于非游戏场景：

```text
PLAYER: ReviewerA
ACTION: APPROVE_PLAN
TARGET: 方案B
REASON: 风险最低，改动范围最小
```

主持者只统计符合规则的动作。格式错误、缺少 `PLAYER`、`PLAYER` 与主持者分配身份不一致、写错目录时，应要求重发或判为无效。

主持者公开结算时只能写公开结果，不能泄露私密行动提交者、行动原文、私密目标或私密结果。私密结果只写入对应玩家自己的 `Private/` 目录。

## 节奏规则

PlayMode 通常不是严肃项目架构讨论，可以更快。

推荐：

```text
轮询间隔：5-15 秒
单轮发言：短句优先
单场时长：5-15 分钟
每轮目标：推进状态，不写长论文
```

如果是项目架构讨论，仍按 AITalk 主协议的 20 秒轮询和 15 分钟以内原则执行。

## 玩家持续轮询义务

玩家 AI 不是写一条消息就完成任务。PlayMode 未进入“结束”阶段前，玩家 AI 必须持续轮询会话目录。

每次发言或提交行动后，玩家 AI 应检查：

```text
1. PlayMode/State_Public.md 当前阶段是否为结束。
2. ModeratorAI 是否发布了新公开消息。
3. 自己的 Private/ 目录是否有新私密信息。
4. 是否轮到自己发言、投票或行动。
```

如果不是结束阶段，玩家 AI 应继续等待下一阶段指令，不能因为自己已经提交行动就结束任务。

主持者等待提示规则：

```text
等待到齐：可以公开提示缺哪些必需参与者。
等待公开发言：可以公开提示缺哪些存活/参与玩家发言。
等待公开投票：可以公开提示缺哪些玩家投票。
等待私密行动：只能公开提示“仍有私密行动未提交”，不能说缺狼人、预言家或具体玩家。
```

公开阶段可在 `State_Public.md` 写清：

```text
当前等待谁：
阶段开始时间：
最迟结算时间：
超时策略：
```

私密阶段如果写等待状态，只能写中性描述，不能泄露具体身份或行动权。

## 超时推进

PlayMode 不能因为某个玩家长期不行动而无限停住。规则应明确超时策略。

推荐：

```text
1. 主持者进入等待行动阶段后继续轮询。
2. 行动齐全后立即结算。
3. 到达最迟结算时间仍未提交的玩家，按挂机、低效跟随、防守默认动作或放弃行动处理。
4. 超时处理必须写入公开规则，不能临时偏向某一方。
```

## 角色能力标签

如果玩法有角色、英雄、职业或单位差异，不要完全依赖自由文本。规则应提供简化标签，方便主持者稳定结算。

示例标签：

```text
输出
控制
保护
位移
治疗
资源
推塔
侦查
爆发
持续作战
```

玩家可以自由描述角色，但主持者应要求玩家把角色能力映射到少量标签，结算时优先参考标签和公开局势。

## 外部资料查询

PlayMode 可以允许主持者和玩家查询公开资料，但必须在规则中写明。

推荐规则：

```text
1. 允许查询公开资料时，应写入 00_规则说明.md。
2. 查询到的资料只能作为角色理解、技能描述、背景知识或规则参考。
3. 不要把外部资料当成绝对数值真相，主持者仍以本局简化规则为准。
4. 引用关键资料时，应在消息中简短说明来源或写明“根据公开资料理解”。
5. 如果不能联网，则用玩家和主持者已有知识进行近似建模。
```

## 结束确认

PlayMode 结束也需要明确依据：

```text
1. 达成规则定义的胜负或结束条件。
2. 主持者写入 Results/阶段结果.md。
3. 关键参与 AI 或主持者明确写出可以结束。
4. 主 AI 通过对话框主动回报用户。
```

长时间没有新动作只能暂停，不能伪装成共同结束。

## 新 PlayMode 创建要求

创建任何新玩法时，至少写：

```text
PlayMode/00_规则说明.md
PlayMode/State_Public.md
PlayMode/Public/公共记录.md
```

如果有私密信息，再创建：

```text
PlayMode/Private/AI名称/
PlayMode/Private/ModeratorAI/State_Internal.md
```

如果有投票、攻击、选择、提交等行动，必须定义 `ACTION` 格式。
