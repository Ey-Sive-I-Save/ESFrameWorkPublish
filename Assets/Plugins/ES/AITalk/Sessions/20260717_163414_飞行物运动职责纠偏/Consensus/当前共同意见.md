# 当前共同意见

## 已达成

```text
架构侧与运动实现侧均同意：飞行物运动模块不应成为飞行物全包系统。
运动模块应聚焦运动模型、运动状态、Tick 更新、到达/失效结果。
输入、技能释放、伤害、Buff、VFX、音效、对象池和全局调度不应进入运动模块核心职责。
运动只是 Entity 或 Item 的能力，不应被提升成独立世界对象身份。
当前倾向：飞行物属于 Item 体系，运动能力落在 ItemBasicDomain 下的 Module，而不是新增一堆 Item Domain。
第一阶段候选结构：ItemBasicDomain 下放 ItemMotionModule 与 ItemProjectileModule；MotionSolver 保持纯 C#，不做 MonoBehaviour。
MotionSolver 不直接写 Transform / Rigidbody，只输出结果。
ProjectileModule 可以生成 HitCandidate，但不执行伤害、Buff、VFX、音效或对象池回收。
同一帧只允许一个权威位姿写回者；当前候选为 ItemMotionModule 统一写回 Transform/Rigidbody。
ItemProjectileModule 负责飞行物语义、调用 MotionSolver、生成 ProjectileMotionResult 和 ProjectileHitCandidate。
MotionResultKind 表达主运动状态，HitCandidate 作为独立字段，不塞进主状态枚举。
ProjectileModule 自持 NonAlloc 查询缓冲，Tick 中不扩容；容量不足时截断并记录调试信息。
目录策略候选：第一阶段优先放在 Item 内部，不急着创建 Runtime/Movement 根目录；复用证据出现后再上提。
```

## 初始架构立场

```text
飞行物运动模块应先明确职责边界，再推进代码扩展。
它不应该同时承担输入、技能决策、特效音效编排、伤害结算、对象池生命周期和全局调度。
它应该优先负责“飞行物如何从当前状态移动到目标状态”的稳定、可复用、可测试逻辑。
```

## 仍有分歧

```text
尚未最终定稿。
Item 的具体代码结构仍需在用户确认后推进。
Projectile 与普通 ItemMotion 的关系当前候选为：ItemProjectileModule 调用 MotionSolver，并通过 ItemMotionModule 或 Item 底层驱动写回位姿。
HitCandidate、MotionResult 字段已有第一版候选，但仍需架构侧最终审阅。
是否第一阶段就实现 ItemLifetimeModule 仍未确认；当前倾向后置。
目录位置仍是候选：真正改代码前必须先扫描项目现有 Item / Movement / Motion / Shared 相关结构，避免新建不自然目录。
最终结论尚未写入；等待架构侧整理候选版或继续审阅。
```

## 风险

```text
如果运动模块边界不清，会很容易变成战斗逻辑、技能逻辑、特效逻辑和对象生命周期的混合体。
```

## 建议执行方案

```text
继续让运动实现侧补充最小可执行结构。
若后续允许改代码，优先把 Item 侧运动结构纠偏为 ItemBasicDomain + ItemMotionModule / ItemProjectileModule。
不要提前扩出很多 Item Domain。
第一阶段只做低风险骨架，不直接接入伤害、VFX、对象池策略。
最小验证链路：直线飞行、超时 Expired、到达 Arrived、Collider 命中候选 hasHitCandidate。
写代码前使用不越界检查清单：不新增很多 Domain、不双写位姿、Solver 不做 MonoBehaviour、不在 Tick 中分配、不做伤害/Buff/VFX/音效/对象池策略。
```
