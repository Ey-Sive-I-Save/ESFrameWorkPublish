# 当前共同意见

## 已达成

- Codex 已提出初稿，架构AI已加入并完成第一轮审查。
- 本会话不改场景、不改 prefab、不改业务代码，只在 AITalk Session 内讨论和沉淀结论。
- 讨论对象不是单一玩家对象，而是高性能、可迁移的所有生命体兼容模型。
- 现阶段倾向采用“窄核心 + 能力组合 + 适配层”，而不是单一巨型继承树。
- `MotionBody` 更适合作为可组合运动能力/适配层，不应与 `LifeActor / CharacterActor` 平级成为完整 Actor 类型。
- 项目已有 `Item : Core`、`ItemBasicDomain`、`ItemMotionModule`、`ItemProjectileModule`，投射物/Item 有独立 Core 路线，不应强行继承 `CharacterActor`。
- 当前 `StateMachine` 可以作为迁移期 `StateAdapter`，但它强绑定 `Entity`，中期应抽成 `IStateMachineHost / ICharacterStateAdapter`。
- `ControlAuthorityResolver` 不应放入 `EntityStateDomain`，更适合放在 `ActorCore / CharacterActorFacade` 层。
- `ESMotionIntent` 应保留为运动驱动层请求；角色语义层需要另设 `MovementIntent`，再转换到 `ESMotionIntent`。
- 最终大方向确认：采用 `ActorCore / LifeActorFacade / CharacterActorFacade / MotionBody / ExternalAdapters`，但实现阶段不急于创建大量实体类，可先以接口、facade 和 adapter 形式落地。
- `Projectile/Item` 不使用完整 `Root_Actor`，只接共享的 Owner/Camp/MotionBody/HitCandidate/DamageSource/Target 协议。
- 迁移路线确认：先 facade 兼容现有 `Entity`，再接口化 State/Motion/Input 强绑定，最后让新 prefab 按 Actor 模板搭建。

## 初步候选共识，待架构AI最终确认

- 玩家应被视为 `ControlAuthority = LocalPlayer` 的生命体实例，而不是独立于生命体体系之外的特殊架构。
- 现有 `Entity` 不宜一次性推倒；更可控路线是先建立上层 `LifeActor / CharacterActor` facade 和协议层，再逐步迁移 `Entity` 的 KCC、状态、战斗、输入职责。
- KCC、InputSystem、FinalIK、Cinemachine、EasySave3、DOTween 应通过适配层进入角色体系，不应成为核心协议类型。
- `ESMotionTypes` 已经具备泛化运动协议雏形，应作为所有运动体兼容的关键入口之一。
- `ControlAuthority` 不应只是简单栈，候选方案是 token + priority + mask 的 `ControlAuthorityResolver`。
- Snapshot 应区分 Runtime / Save / Network，不应强行合成一个万能结构。
- 最终命名候选：`ActorCore / LifeActorFacade / CharacterActorFacade / MotionBody / ExternalAdapters`。
- `Projectile/Item` 不使用完整 `Root_Actor`，只接共享的 Owner/Camp/MotionBody/HitCandidate/DamageSource/Target 协议。

## 仍有分歧

- 命名基本确认，但需要在最终结论中强调“先作为架构概念和接口边界，不要求第一阶段全部新建类”。
- Unity 层级模板需要最终收束为“基础稳定节点 + variant 增量”，避免所有对象复制同一套大层级。

## 风险

- 抽象层过多可能导致落地慢、调试困难。
- 如果不控制边界，`EntityBasicModules.cs` 会继续膨胀。
- 如果角色切换、剧情控制、网络控制都直接写 Transform，会破坏运动和战斗权威链。
- 如果所有对象强行继承同一完整角色核心，会让投射物、载具、道具承担不必要成本。

## 建议执行方案

- 先讨论并确认通用分层：Identity、Authority、Intent、Domain、Motion、State、Presentation、Snapshot。
- 再确认 Unity 层级模板：稳定节点、可选节点、运行时生成节点。
- 最后给出逻辑链验证表，覆盖 MMO、开放世界、角色切换、剧情、RPG 战斗、NPC/怪物/Boss、召唤物/宠物、载具、投射物。
- 迁移路线分三阶段：先 facade 兼容现有 `Entity`，再接口化 State/Motion/Input 强绑定，最后让新 prefab 按 Actor 模板搭建。
