## 结论摘要

本项目不应继续做“玩家对象专用模板”，而应建立“所有生命体兼容”的高性能 Actor 模型。玩家只是 `ControlAuthority = LocalPlayer` 的一个生命体实例；NPC、怪物、Boss、宠物、召唤物、剧情演员、远程玩家、可切换角色都应共享核心协议和生命周期，但通过能力组合决定具体功能。

最终架构方向确认：

```text
ActorCore
LifeActorFacade
CharacterActorFacade
MotionBody
ExternalAdapters
```

第一阶段不要求这些全部成为真实 C# 类。商业级落地应先用少量接口、facade、adapter 和验证链路建立边界，避免为了抽象完整而创建大量空类型。

## 与现有 Entity 的关系

现有 `Entity` 不应一次性推倒。

第一阶段：

```text
CharacterActorFacade
  -> 缓存 Entity、StateMachine、Animator、KCC、关键 Domain/Module
  -> 接收 CharacterIntentFrame
  -> ControlAuthorityResolver 过滤控制权
  -> 调用 Entity.SetMoveInput / RequestJump / StateMachine / Skill 等现有 API
```

第二阶段：

```text
Entity -> ICharacterMotionHost / ICharacterStateHost / ICharacterPresentationHost
StateMachine.Entity host -> IStateMachineHost / IActorStateContext
KCC direct API -> IMotionDriverAdapter
InputBridge -> IntentSource
```

第三阶段：

```text
新生命体 prefab 直接按 Actor 模板搭建
旧 Entity prefab 继续通过 facade 兼容
```

## 控制权方案

`ControlAuthorityResolver` 采用 token + priority + mask，不使用简单栈，不放进 `EntityStateDomain`。

```text
AuthorityToken
  -> ControlAuthorityResolver
  -> IntentFilter
  -> MovementIntent / SkillIntent / InteractionIntent
  -> MotionAdapter / StateAdapter / SkillAdapter
```

示例 token：

```text
LocalInputToken
AIToken
NetworkToken
CutsceneToken
StunToken
VehicleToken
ReplayToken
```

`StateMachine` 可以表达眩晕、霸体、禁用技能等状态结果，但不应决定谁拥有角色控制权。

## 运动方案

`ESMotionIntent` 保留为运动驱动层请求，上层另设 `MovementIntent`。

```text
MovementIntent
  -> CharacterMovementPolicy / LocomotionResolver
  -> ESMotionIntent
  -> IESMotionDriver.ApplyMotionIntent
  -> ESMotionState
```

KCC 只是 MotionBody 的一种 driver，不应污染输入、剧情、网络、AI 的上层协议。

## StateMachine 方案

当前 `StateMachine` 可作为迁移期 `StateAdapter`，但它强绑定 `Entity`：

```text
StateMachine.hostEntity: Entity
StateMachine.Initialize(Entity entity, ...)
EntityState_Skill
```

因此后续应逐步抽成：

```text
ICharacterStateAdapter
IStateMachineHost
IActorStateContext
```

不建议立刻重写状态机，否则会破坏已有技能、动画和 IK 链路。

## 高性能硬约束

- 热路径不允许每帧 `GetMoudle<T>()`、`Find`、反射、字符串查找。
- facade 初始化时缓存 `Entity`、`StateMachine`、KCC、Animator、Domain/Module 强类型引用。
- `CharacterIntentFrame` 使用 struct 或复用缓存，不每帧 new class/list。
- Adapter 只做边界转换，不在 Tick 内做复杂查表。
- Snapshot 按 Runtime / Save / Network 拆分，不设计万能快照。
- `Core -> Domain -> Module` 适合组织结构，但不代表热路径可以每帧动态查模块。

## 完整逻辑链验证

本地玩家：

```text
InputSystem
  -> LocalInputIntentSource
  -> AuthorityResolver(LocalInputToken)
  -> MovementIntent / SkillIntent
  -> Entity / KCC / StateMachine
  -> Animator / IK / CameraTarget
  -> RuntimeSnapshot
```

远程玩家：

```text
NetworkSnapshot
  -> NetworkAuthorityToken
  -> MotionSmoothingAdapter
  -> StateAdapter
  -> Presentation
  -> NetworkRuntimeSnapshot
```

NPC / 怪物：

```text
AIPlanner
  -> AIIntentSource
  -> AuthorityResolver(AIToken, StunToken...)
  -> Motion / Combat / Skill
  -> Threat / Target
  -> State / Presentation
```

剧情演员：

```text
Timeline / Cutscene
  -> CutsceneAuthorityToken
  -> MovementIntent 或 DirectMotionRequest
  -> MotionAdapter
  -> DialogueLink / Presentation
```

角色切换：

```text
Party/System
  -> AuthorityToken 转移
  -> 旧角色 LocalInputToken 失效 / AI 接管
  -> 新角色 LocalInputToken 生效
  -> CameraTarget 切换
```

Boss：

```text
BossAI + PhaseModule
  -> AuthorityResolver
  -> MultiPartHit / Threat / Skill
  -> BossStateAdapter
  -> Presentation / WeakPoint / PhaseSnapshot
```

宠物 / 召唤物：

```text
OwnerCommand + PetAI + LeashRule
  -> AuthorityResolver
  -> Motion / Combat
  -> OwnerLink Snapshot
```

载具：

```text
Character MountRequest
  -> Vehicle SeatLink + VehicleAuthorityToken
  -> Character motion 部分禁用
  -> Vehicle MotionBody 驱动
  -> Passenger Presentation 同步
```

投射物：

```text
Skill / Weapon
  -> ItemProjectileModule 或 ProjectileMotion
  -> MotionBody
  -> HitCandidate
  -> DamageSource
  -> 目标 LifeActor / CombatAdapter
```

## 执行建议

可以确认架构方向，可以准备第一阶段 facade 验证。

不建议：

- 立刻大规模替换 `Entity`。
- 立刻改场景层级。
- 为了命名完整创建大量空类。
- 把投射物、载具、Boss 特性塞进 CharacterActor 基类。
- 继续扩大 `EntityBasicModules.cs`。

下一步最合理的工程动作：

1. 设计第一阶段接口和 facade 最小集。
2. 选一个现有 Entity prefab/场景模板做只读链路验证。
3. 写一份 Actor 层级模板说明或生成工具草案。
4. 再决定是否把本结论长期沉淀到 `Assets/Plugins/ES/AIWarnings`。
