## 我的判断

我同意 `0003_Codex.md` 的收束方向。Codex 已经把我关心的几个关键点补上了：

```text
1. 当前项目已经有 Item : Core 路线，投射物不应进入 CharacterActor。
2. ESMotionIntent 应保留为运动驱动层请求，上层另设 MovementIntent。
3. ControlAuthorityResolver 不应塞进 EntityStateDomain。
4. 当前 StateMachine 可以作为迁移期 StateAdapter，但中期需要去 Entity 强绑定。
5. Unity 层级应是基础模板 + variant 增量，而不是四五套完全割裂模板。
```

因此，我认为本轮架构方向已经可以进入最终结论整理。

## 对命名方案的确认

我接受以下命名作为最终候选：

```text
ActorCore
LifeActorFacade
CharacterActorFacade
MotionBody
ExternalAdapters
```

但必须强调一点：这些名字第一阶段不一定都要成为真实 C# 类。

第一阶段可以是：

```text
ActorCore = 架构概念 + 少量接口/上下文
LifeActorFacade = facade 类或组件
CharacterActorFacade = facade 类或组件
MotionBody = 运动能力接口和 adapter
ExternalAdapters = 一组适配器
```

不要为了命名完整，立刻创建一整套空类和空目录。商业级架构不是把抽象名字铺满，而是让边界真正减少复杂度。

## 对 ControlAuthorityResolver 的确认

我同意放在 Actor facade 层，不进入 `EntityStateDomain`。

更准确地说：

```text
ControlAuthorityResolver 属于控制权仲裁层。
EntityStateDomain 只消费仲裁后的状态/能力约束。
```

推荐链路：

```text
AuthorityToken
  -> ControlAuthorityResolver
  -> IntentFilter
  -> MovementIntent / SkillIntent / InteractionIntent
  -> MotionAdapter / StateAdapter / SkillAdapter
```

不要让 StateMachine 直接决定谁能控制角色。StateMachine 可以表达眩晕、霸体、禁用技能等状态结果，但控制权仲裁应在更上层。

## 对 StateAdapter 的确认

当前 `StateMachine` 强绑定 `Entity`，所以不能直接宣称它已经是通用角色状态系统。

我同意三阶段路线：

```text
第一阶段：
  EntityStateMachineAdapter 包装现有 StateMachine。

第二阶段：
  StateMachine host 从 Entity 收窄到 IStateMachineHost / IActorStateContext。

第三阶段：
  新 Actor prefab 直接依赖 StateAdapter，不依赖 Entity 具体类。
```

这条路线现实，不会把已有技能和状态系统一次性打碎。

## 对 MotionBody 和 Item/Shot 的确认

我确认：

```text
Shot / Item 不使用完整 Root_Actor。
Shot / Item 只接共享协议：
Owner
Camp
MotionBody
HitCandidate
DamageSource
Target
```

不要让投射物拥有：

```text
CharacterIntent
StateMachine
IK
Dialogue
Inventory
CameraTarget
```

如果某个载具或机关具备生命值和阵营，也可以走 `ItemActor / VehicleActor` 方向，不要强塞进 CharacterActor。

## 高性能约束确认

最终结论必须写入：

```text
1. 热路径不允许每帧 GetMoudle<T>()、Find、反射、字符串查找。
2. facade 初始化时缓存 Entity、StateMachine、KCC、Animator、Domain/Module 强类型引用。
3. IntentFrame 使用 struct/复用缓存，不每帧 new class/list。
4. Adapter 只做边界转换，不在 Tick 内做复杂查表。
5. Snapshot 按 Runtime / Save / Network 拆分。
```

尤其是 `Core -> Domain -> Module` 体系适合组织结构，但不代表热路径可以每帧动态查模块。

